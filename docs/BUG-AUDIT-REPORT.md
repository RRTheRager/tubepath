## Critical issues (fix first)

### 1. Checkout success session hijack

**File:** `src/app/api/billing/checkout-success/route.ts:43-45`

After Stripe checkout completes, if the browser’s session cookie doesn’t match the paying `accountId`, the handler calls `setSessionId(accountId)`. Anyone with a leaked `session_id` (history, Referer, screenshot) can attach a victim’s paid account to their browser.

**Fix:** Store a signed `checkout_state` cookie at checkout start; on return, verify it matches before switching sessions. Never reassign sessions from Stripe metadata alone.

---

### 2. Billing simulator + free trial in production without Stripe

**Files:** `src/app/api/billing/simulate/route.ts`, `src/app/api/billing/start-trial/route.ts`

Both routes only check `isStripeConfigured()`. If Stripe env vars are missing/misconfigured in production:

- `POST /api/billing/simulate` → set status to `"active"` for free
- `POST /api/billing/start-trial` → start trial with no card

**Fix:** Fail closed in production (`NODE_ENV === "production"`) regardless of Stripe config. Keep simulator dev-only behind `ENABLE_BILLING_SIMULATOR=true`.

---

### 3. Trial auto-promotes to paid `active` without Stripe confirmation

**File:** `src/lib/store.ts:60-69` (`reconcile()`)

On every `getAccount()` read, an expired `trialing` account becomes `active` with a synthetic 30-day period — no payment verification. If a webhook is delayed or missed, users get full premium access for free.

**Fix:** Skip auto-promotion when `account.stripeSubscriptionId` is set. Only simulate locally when Stripe is not configured.

---

### 4. Supabase writes silently fail

**File:** `src/lib/storage/supabase.ts`

All `upsert`/`delete` calls destructure `{ data }` without checking `error`. Failed writes (RLS, constraints, network) appear successful; subscriptions and OAuth tokens may not persist.

**Fix:** Check `error` on every Supabase call; throw and propagate to API responses.

---

### 5. Conflicting database schemas

**Files:** `supabase/schema.sql` vs `supabase/migrations/20250627000000_initial_schema.sql`

The app expects `public.accounts` (text PK) and text-keyed `google_accounts` (`schema.sql`). The initial migration defines `profiles` + UUID `google_accounts` tied to `auth.users` — a different product model. Migrations `002`/`003` alter tables that the initial migration never creates.

**Fix:** Replace the obsolete initial migration with one matching `schema.sql`, or delete migrations and document “run schema.sql once” exclusively.

## Medium issues

- **Ephemeral sessions in Server Components** — `getSessionId()` creates IDs that may not persist (`session.ts:38-46`)
- **`maxChannels` capability never enforced** — trial users can switch all channels (`access.ts`)
- **Analytics `colIndex` returns -1 silently** — wrong metrics from `row[-1]` (`youtube.ts`, `studio-analytics.ts`, `pipeline/build.ts`)
- **`updateVideo` partial snippet** — may clear YouTube fields (`data/youtube.ts`)
- **No rate limiting** on AI / sync endpoints — cost exhaustion risk
- **No `error.tsx` boundaries** — single failure crashes app shell
- **Pipeline errors leak internal messages** — `src/app/api/pipeline/route.ts`
- **Legacy monorepo drift** — `apps/web` demo mode defaults ON; workspaces not configured
- **2 moderate npm audit findings** — postcss XSS advisory via next dependency tree

## Architecture improvement roadmap

### Phase 1 — Security & billing (1–2 focused PRs)

1. Secure checkout-success flow (signed state cookie)
2. Harden sessions (CSPRNG + HMAC cookies)
3. Production fail-closed for simulate/start-trial
4. Fix `reconcile()` Stripe bypass
5. Supabase error handling on all writes
6. Remove or gate `/api/ai/health`

### Phase 2 — Data integrity

1. Unify Supabase schema/migrations
2. Immediate local sync on Stripe cancel/resume
3. Set `graceEndsAt` from webhooks for `past_due`
4. Encrypt OAuth refresh tokens at rest
5. Add account update locking (version column or RPC)

### Phase 3 — UX honesty & resilience

1. Remove fake anomaly placeholders; show real empty states
2. Fix GoogleConnectGate + SessionProvider error handling
3. Add `error.tsx` at app segments
4. Enforce `maxChannels` in store + API
5. Add rate limiting (Upstash or middleware) on AI/sync routes

### Phase 4 — Codebase hygiene

1. **Decide:** delete `apps/` + `packages/` OR wire npm workspaces + separate CI
2. Consolidate duplicated YouTube helpers (`num`, `colIndex`, channel fetch) into one module
3. Consolidate duplicate AI insight prompts (`ai/index.ts` vs `ai/analytics.ts`)
4. Add security headers in `next.config.ts` (CSP, X-Frame-Options, Referrer-Policy)
5. Add integration tests for billing webhook + checkout flow
